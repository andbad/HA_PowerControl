# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Development workflow commands

- **Installation / reconstruction**
  - The component is a Home Assistant custom integration. To install locally for development:
    ```bash
    pip install -e .
    ```
  - If using HACS, add the repository URL `https://github.com/andbad/HA_PowerControl` as a custom repository and install via HACS UI.

- **Running tests**
  - Tests are located under the `tests/` directory and use pytest. Run all tests with:
    ```bash
    pytest
    ```
  - To run a single test file:
    ```bash
    pytest tests/test_<file>.py
    ```

- **Linting / formatting**
  - This project uses flake8 for linting and black for code formatting. Run linters with:
    ```bash
    flake8 .
    black --check .
    ```

- **Building / packaging**
  - No additional build step is required; the integration is loaded by Home Assistant at runtime.
  - When distributing via HACS, the component folder should contain the `custom_components/power_control` package.

## Code architecture overview

- **Package structure**
  - `custom_components/power_control/` – main source tree of the integration.
  - `manifest.json` (generated by HACS) – component metadata.
  - `configuration.yaml` – user configuration is defined in the UI wizard; no yaml file is stored in the repo.
  - `README.md` – user-facing documentation (see top-level README).
  - `MIGRATION.md` – migration guide for breaking changes.
  - `tests/` – pytest unit and integration tests.

- **Key modules**
  - `configuration.py` – defines the configuration flow, wizard steps, and default options.
  - `const.py` – constants and default values (e.g., thresholds, delays).
  - `entities.py` – defines custom sensor and switch entities exposed to Home Assistant.
  - `services.py` – implements the service calls (`enable`, `disable`, `reset_load`, etc.).
  - `dashboard.py` – creates the Lovelace dashboard automatically when the user opts in.
  - `utils.py` – helper functions for power calculations and state management.

- **Entity lifecycle**
  - Two modes of disconnection: **immediate** and **delayed**, each with configurable thresholds and timers.
  - Loads are prioritized; the system turns off loads in reverse priority order and turns them back on in forward priority order, respecting cooldown periods.
  - The global power sensor (`sensor.power_control_potenza_attuale`) reports current consumption; suspended loads are aggregated in `sensor.power_control_potenza_sospesa`.

- **Dashboard**
  - Automatically generated Lovelace dashboard includes a power gauge, real‑time status, history graph, configuration card, and per‑load cards.
  - Dashboard files are created under the `www/` directory of the component; no manual import required.

## Frequently asked questions for future Claude Code users

1. **How do I add a new load?**  
   Edit the configuration wizard (step 2 in the README) or manually add an entry in `configuration.yaml` if you prefer declarative config. Remember to update the priority order; load index `0` is highest priority.

2. **Which sensor should I use for global power?**  
   Prefer a physical whole‑house sensor (e.g., Shelly EM). If unavailable, the component can sum the individual load powers, but thresholds should be set conservatively.

3. **Can I disable automatic reactivation?**  
   Yes – set the *Riattivazione automatica* flag to *Disabled* for a given load; the load will stay off until manually started via the `power_control.force_start_load` service.

4. **How are the service calls exposed?**  
   Services are registered under the `power_control` domain (`power_control.enable`, `power_control.disable`, `power_control.reset_load`, `power_control.force_stop_load`, `power_control.force_start_load`).

By keeping these commands and architectural points in mind, future Claude Code sessions should be able to navigate, modify, and extend the Power Control integration efficiently.
